# 前端登录注册模块接口说明

本文档面向前端开发，说明当前后端已经实现的登录、注册、找回密码、当前用户与登出接口。

## 1. 基础说明

### 1.1 接口基础路径

所有认证接口统一挂载在：

```text
/api/v1/auth
```

### 1.2 统一响应格式

所有接口都返回统一 JSON 结构：

```json
{
  "code": 0,
  "message": "登录成功",
  "data": {},
  "timestamp": 1750000000000,
  "requestId": "xxx"
}
```

字段说明：

- `code`
  - `0` 表示成功
  - 非 `0` 表示失败
- `message`
  - 成功或失败描述
- `data`
  - 业务数据
  - 无返回数据时为 `null`
- `timestamp`
  - 服务端响应时间戳，毫秒
- `requestId`
  - 请求追踪 id，便于排障

### 1.3 常见错误码

| code | HTTP 状态码 | 含义 |
| --- | --- | --- |
| `40000` | `400` | 参数错误 |
| `40100` | `401` | 未登录或登录已过期 |
| `40300` | `403` | 无权限访问该资源 / CSRF 校验失败 |
| `40310` | `403` | 当前时期不允许该操作 |
| `40400` | `404` | 资源不存在 |
| `40900` | `409` | 数据冲突 |
| `40910` | `409` | 业务状态冲突 |
| `42200` | `422` | 数据校验失败 |
| `42900` | `429` | 请求过于频繁 |
| `50000` | `500` | 服务器内部错误 |

### 1.4 登录态与 Cookie

当前系统使用 `HttpOnly Cookie + CSRF Token` 维护登录态，不是前端手动保存 JWT。

登录成功后，后端会写入两个 Cookie：

- `lab_recruit_token`
  - 登录态 Cookie
  - `HttpOnly`，前端 JS 不能读取
- `XSRF-TOKEN`
  - CSRF Cookie
  - 前端 JS 可以读取

### 1.5 前端请求注意事项

- 前端请求必须开启 `withCredentials`
- 登录后，所有“已登录且会修改状态”的接口都必须带上请求头：

```text
X-CSRF-TOKEN: <XSRF-TOKEN 的值>
```

- 以下公开接口不需要带 `X-CSRF-TOKEN`
  - `POST /api/v1/auth/login`
  - `POST /api/v1/auth/register`
  - `POST /api/v1/auth/send-email-code`
  - `POST /api/v1/auth/forgot-password`
  - `POST /api/v1/auth/reset-password`

- 以下接口需要登录：
  - `POST /api/v1/auth/logout`
  - `GET /api/v1/auth/me`
  - `POST /api/v1/auth/change-password`

## 2. 枚举值

### 2.1 验证码场景 `scene`

| 值 | 含义 |
| --- | --- |
| `REGISTER` | 注册验证码 |
| `RESET_PASSWORD` | 重置密码验证码 |

### 2.2 角色 `role`

| 值 | 含义 |
| --- | --- |
| `FRESHMAN` | 普通新生用户 |
| `LEADER` | 组长 |
| `ADMIN` | 管理员 |

### 2.3 用户状态 `status`

| 值 | 含义 |
| --- | --- |
| `ACTIVE` | 正常 |
| `DISABLED` | 禁用 |

## 3. 接口列表

## 3.1 发送验证码

### 接口

```http
POST /api/v1/auth/send-email-code
Content-Type: application/json
```

### 请求体

```json
{
  "email": "user@example.com",
  "scene": "REGISTER"
}
```

### 参数说明

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `email` | `string` | 是 | 邮箱 |
| `scene` | `string` | 是 | `REGISTER` 或 `RESET_PASSWORD` |

### 成功响应

```json
{
  "code": 0,
  "message": "验证码已发送",
  "data": null,
  "timestamp": 1750000000000,
  "requestId": "xxx"
}
```

### 业务说明

- 注册场景下：
  - 必须在报名期内
  - 邮箱不能已注册
- 找回密码场景下：
  - 即使邮箱不存在，也会返回统一成功文案，前端不要依赖它判断邮箱是否存在
- 验证码默认 5 分钟有效
- 发送过于频繁会返回 `42900`
- 验证码错误次数过多后，短时间内会被锁定，需要重新获取

---

## 3.2 注册

### 接口

```http
POST /api/v1/auth/register
Content-Type: application/json
```

### 请求体

```json
{
  "username": "fresh_user",
  "email": "user@example.com",
  "password": "Pass1234",
  "confirmPassword": "Pass1234",
  "code": "123456"
}
```

### 参数说明

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `username` | `string` | 是 | 用户名，3 到 32 位，仅允许字母、数字、下划线 |
| `email` | `string` | 是 | 邮箱 |
| `password` | `string` | 是 | 密码 |
| `confirmPassword` | `string` | 是 | 确认密码 |
| `code` | `string` | 是 | 6 位验证码 |

### 成功响应

```json
{
  "code": 0,
  "message": "注册成功",
  "data": {
    "id": 1,
    "username": "fresh_user",
    "email": "user@example.com",
    "role": "FRESHMAN"
  },
  "timestamp": 1750000000000,
  "requestId": "xxx"
}
```

### 业务说明

- 必须在报名期内
- 用户名和邮箱都必须唯一
- 密码规则：
  - 至少 8 位
  - 必须同时包含字母和数字
- 两次密码必须一致
- 当前注册成功后不会自动登录
- 当前实现中，注册用户邮箱会直接标记为已验证

---

## 3.3 登录

### 接口

```http
POST /api/v1/auth/login
Content-Type: application/json
```

### 请求体

```json
{
  "email": "user@example.com",
  "password": "Pass1234",
  "rememberMe": false
}
```

### 参数说明

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `email` | `string` | 是 | 邮箱 |
| `password` | `string` | 是 | 密码 |
| `rememberMe` | `boolean` | 否 | 是否记住登录，缺省视为 `false` |

### 成功响应

```json
{
  "code": 0,
  "message": "登录成功",
  "data": {
    "id": 1,
    "username": "fresh_user",
    "email": "user@example.com",
    "role": "FRESHMAN",
    "status": "ACTIVE",
    "emailVerified": true
  },
  "timestamp": 1750000000000,
  "requestId": "xxx"
}
```

### 业务说明

- 登录成功后，后端会自动写入：
  - `lab_recruit_token`
  - `XSRF-TOKEN`
- 不需要前端手动保存 JWT
- 登录失败会返回统一文案，不区分“密码错 / 邮箱不存在 / 账号禁用”
- 同一邮箱在短时间内连续失败过多会被限流

## 3.4 登出

### 接口

```http
POST /api/v1/auth/logout
```

### 请求头

```text
X-CSRF-TOKEN: <XSRF-TOKEN 的值>
```

### 成功响应

```json
{
  "code": 0,
  "message": "退出成功",
  "data": null,
  "timestamp": 1750000000000,
  "requestId": "xxx"
}
```

### 业务说明

- 需要登录
- 需要带 `X-CSRF-TOKEN`
- 登出后后端会清除认证 Cookie

## 3.5 获取当前登录用户信息

### 接口

```http
GET /api/v1/auth/me
```

### 成功响应

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "id": 1,
    "username": "fresh_user",
    "email": "user@example.com",
    "role": "FRESHMAN",
    "status": "ACTIVE",
    "emailVerified": true,
    "leaderGroupId": null,
    "groups": [
      {
        "id": 10,
        "name": "前端组"
      }
    ]
  },
  "timestamp": 1750000000000,
  "requestId": "xxx"
}
```

### 字段说明

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `number` | 用户 id |
| `username` | `string` | 用户名 |
| `email` | `string` | 邮箱 |
| `role` | `string` | 角色 |
| `status` | `string` | 用户状态 |
| `emailVerified` | `boolean` | 是否已验证邮箱 |
| `leaderGroupId` | `number \| null` | 如果当前用户是组长，这里是其负责的小组 id，否则为 `null` |
| `groups` | `array` | 当前用户所属小组列表 |

### 业务说明

- 需要登录
- 用于页面刷新后恢复当前登录态

### 参数说明

| 字段 | 类型 | 必填 | 说明 |

| `email` | `string` | 是 | 邮箱 |

| `confirmPassword` | `string` | 是 | 确认新密码 |

### 成功响应


## 3.8 修改密码

### 接口

```http
POST /api/v1/auth/change-password
Content-Type: application/json
```

### 请求头

```text
X-CSRF-TOKEN: <XSRF-TOKEN 的值>
```

### 请求体

```json
{
  "oldPassword": "OldPass123",
  "newPassword": "NewPass123",
  "confirmPassword": "NewPass123"
}
```

### 参数说明

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `oldPassword` | `string` | 是 | 旧密码 |
| `newPassword` | `string` | 是 | 新密码 |
| `confirmPassword` | `string` | 是 | 确认新密码 |

### 成功响应

```json
{
  "code": 0,
  "message": "修改密码成功，请重新登录",
  "data": null,
  "timestamp": 1750000000000,
  "requestId": "xxx"
}
```

### 业务说明

- 需要登录
- 需要带 `X-CSRF-TOKEN`
- 旧密码必须正确
- 两次新密码必须一致
- 密码规则同注册
- 修改成功后：
  - 当前登录态会失效
  - 前端应跳回登录页

## 4. 前端接入建议

### 4.1 Axios/fetch 必须带 Cookie

示例：

```ts
axios.create({
  baseURL: "/api/v1",
  withCredentials: true
})
```

### 4.2 已登录的写操作自动带上 CSRF 头

前端读取 `XSRF-TOKEN` Cookie 后，在以下请求里带上：

- `POST /api/v1/auth/logout`
- `POST /api/v1/auth/change-password`

请求头：

```text
X-CSRF-TOKEN: <XSRF-TOKEN 值>
```

### 4.3 页面初始化建议

前端应用启动后建议：

1. 调用 `GET /api/v1/auth/me`
2. 如果成功，恢复用户信息到全局状态
3. 如果返回 `40100`，视为未登录

### 4.4 推荐的前端流程

- 注册流程
  1. 调用 `POST /api/v1/auth/send-email-code`
  2. 调用 `POST /api/v1/auth/register`
  3. 注册成功后跳转登录页

- 登录流程
  1. 调用 `POST /api/v1/auth/login`
  2. 后端自动写 Cookie
  3. 调用 `GET /api/v1/auth/me` 或直接使用登录返回数据

- 找回密码流程
  1. 调用 `POST /api/v1/auth/forgot-password`
  2. 调用 `POST /api/v1/auth/reset-password`
  3. 成功后跳转登录页

- 修改密码流程
  1. 已登录状态下调用 `POST /api/v1/auth/change-password`
  2. 成功后清空前端用户状态并跳转登录页
